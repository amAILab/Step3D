# Step3D UX/UI 100+ — 25 автоматических этапов

Цель: не просто “полировать”, а системно поднять интерфейс: убрать конфликтующие слои, усилить mobile-first путь заявки, App, Viewer, карточку заявки, accessibility и release checks.

## 25 этапов

1. CSS layer audit и карта конфликтов.
2. Дизайн-токены и единый premium layer.
3. Главная: один hero-сценарий.
4. App first screen: продуктовая композиция.
5. App форма: меньше полей, яснее обязательное/потом.
6. Project card: номер, статус, next step.
7. Bottom nav: 3 действия, нормальные hit areas.
8. Viewer hero: необязательный путь.
9. Viewer panel: меньше dashboard, больше “сохранить в заявку”.
10. Account preview: карточка заявки, не кабинет.
11. Thanks page: следующий шаг без отвлечений.
12. Trust layer: компактно, без лендингового шума.
13. CTA vocabulary: единые названия кнопок.
14. Typography pass: русские заголовки без слипания.
15. Contrast pass: вторичный текст и dark sections.
16. Accessibility pass 2: forms/menu/modal/tabs.
17. Motion pass: reduced motion и hover без перегруза.
18. Mobile 320/360/390/430 hardening.
19. Tablet layout hardening.
20. Desktop spacing and max-width pass.
21. SEO/schema не ломает UX.
22. Analytics/funnel events consistency.
23. Release checks expansion.
24. Документация дизайн-системы.
25. Финальный deploy-аудит и summary.

## Stage 1 — CSS layer audit

Проблема: UX/UI выглядит слабее, чем должен, не из-за одной кнопки, а из-за каскада старых inline-стилей, mobile-readability, minimal-ui и premium overlay.

Текущий безопасный вывод: полный рефакторинг inline CSS делать надо, но постепенно. Первый правильный шаг — зафиксировать карту конфликтов и дальше выносить правила в системные файлы, не ломая рабочий сайт.


## Stage 2 — Design tokens

Добавлен `assets/step3d-design-system.css` как будущий source of truth:

- цвета: фон, поверхность, текст, muted, line, accent;
- радиусы;
- тени;
- page spacing;
- базовые utility-классы `s3d-surface`, `s3d-chip`, `s3d-grid-*`.

Этот файл пока не ломает существующий сайт, но задаёт базу для постепенного удаления inline CSS и `!important`-слоёв.


## Stage 3 — Homepage hero

Главная усилена как один продуктовый сценарий:

1. Покажите деталь.
2. Получите маршрут.
3. Досылайте файлы с номером S3D.

Добавлен компактный `.home-focus-strip`, который заменяет ощущение “лендинга с кучей доказательств” на понятную последовательность действия.


## Stage 4 — App first screen

App first screen сделан более продуктовым:

- заголовок: `Заявка за минуту. Ответ — по делу.`;
- меньше абстракции, больше конкретики: задача, размер, срок, контакт;
- добавлен promise strip: без чертежа можно, файлы после номера, ответ = маршрут + риски.


## Stage 5 — App form simplicity

Форма разделена по смыслу:

- сейчас: контакт + задача;
- потом: фото/STL/STEP + размеры.

Добавлен блок `.required-later`, уточнены placeholder'ы задачи и файлов.


## Stage 6 — Project card

Карточка проекта стала центром следующего действия:

- добавлен бейдж `Следующий шаг`;
- заголовок `Карточка заявки`;
- добавлен блок `Что будет дальше`: дослать файлы → инженер проверит → даст маршрут/срок/вилку бюджета.


## Stage 7 — Bottom nav

App bottom nav теперь физически содержит только 3 действия:

- Заявка;
- Файлы;
- Статус.

Viewer убран из нижней навигации, потому что это необязательный путь.


## Stage 8 — Viewer hero

Viewer hero переписан как необязательный шаг:

- `Есть STL? Проверьте и отправьте в заявку`;
- явно сказано: Viewer нужен только если уже есть 3D-файл;
- CTA: `Открыть STL/OBJ/3MF`.


## Stage 9 — Viewer panel

Viewer panel стал менее dashboard-heavy:

- `Параметры, если есть модель` → `Что нужно для заявки`;
- `Скопировать расчёт для заявки` → `Сохранить расчёт в заявку`;
- `Оценка` → `Предварительно`;
- следующий шаг стал про сохранение расчёта в заявку Step3D.


## Stage 10 — Account preview

Account wording усилен как честная карточка заявки:

- текст объясняет: финальное подтверждение приходит от менеджера;
- `Создать проект` → `Создать заявку`;
- `frontend-preview` → `preview-карточка`;
- footer объясняет, что приватные файлы и настоящий статус требуют backend/auth.
