## Общий график по аспектам

### День 1 — конверсия и структура
- Главная: сделать путь «у меня есть фото/STL → получить маршрут → оставить заявку» максимально очевидным.
- Мобильная версия: убрать лишний шум, проверить CTA и липкую навигацию.
- Формы: усилить подсказки, шаблоны заявки, готовность заявки.
- Результат: рост понятности и кликов в заявку/Telegram.

### День 2 — услуги, кейсы и доверие
- Сценарии: сломанная деталь, STL по фото, прототип, малая серия, образовательный проект, B2B.
- Кейсы: привести к формату «задача → процесс → результат → что прислать для похожего проекта».
- SEO: закрепить локальные запросы Москва/3D-печать/CAD/STL/реверс.
- Результат: сайт отвечает на вопрос клиента до переписки.

### День 3 — CRM и автоматизация
- Lead pipeline: форма → нормализация → локальный лог → Google Sheet CRM → карточка задачи.
- Проверки: `project_healthcheck`, `validate_site`, `validate_lead_payload`, `check_analytics_events`.
- Контроль публикации: GitHub diff/review → deploy → проверка публичных ссылок.
- Результат: меньше ручной обработки заявок, понятный статус проекта.

### День 4–5 — продуктовый слой
- Личный кабинет / Mini App: сценарий клиента, статусы, файлы, следующий шаг.
- PWA: улучшить app-first вход, shortcuts и офлайн-страницы.
- Контент-план: Telegram-посты, лид-магниты, статьи, FAQ.
- Результат: сайт становится не только лендингом, а входом в сервис.

### Еженедельно
- 1 аудит конверсии и мобильной версии.
- 1 SEO/контент-обновление.
- 1 проверка лидов/CRM/форм.
- 1 коммерческий пакет или кейс.

## Что не делаем без отдельного подтверждения

- Не трогаем старый живой сайт вне preview/public GitHub Pages.
- Не отправляем письма/сообщения клиентам.
- Не публикуем пользовательские файлы/частные фото наружу.
- Не включаем настоящий кабинет/auth без backend и политики хранения файлов.
